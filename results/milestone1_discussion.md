## Discussion / Summary
BM25 strengths and weaknesses: BM25 works best when the query contains specific product terminology that appears directly in product titles — for example, a query like "drum sticks wood tip" maps cleanly onto product listings. Its weakness is that it has no understanding of intent or meaning. Queries phrased as needs ("something to connect my guitar to an amp") or that use synonyms and related concepts will fail because BM25 only counts keyword matches. It is also vulnerable to false positives from partial keyword overlap, as seen when "keyboard" matched a dust cover instead of a piano.
Semantic search strengths and weaknesses: Semantic search handles natural language and intent-based queries much better. It correctly retrieved a cable for a connectivity query and lightweight drumsticks for a wrist fatigue query — connections that require understanding meaning rather than matching words. Its weakness is that it is only as good as the corpus it searches over. When the relevant product category is underrepresented in the dataset (as appears to be the case with digital pianos), semantic search will still return the wrong product type, just with higher confidence.
Queries that are challenging for both methods: Queries that combine multiple constraints simultaneously — skill level, budget, use case, and product type all at once — are hard for both models. Neither BM25 nor a single embedding vector can reliably balance several requirements in one query. Both models also struggled when the corpus lacked sufficient coverage of the target product category, which no retrieval method can compensate for on its own.
Where more advanced methods would help: A reranker would help by rescoring BM25 or semantic candidates against the full query context, catching cases where the top retrieved result is topically adjacent but not actually what the user needs. RAG would help on complex queries by letting a language model reason over multiple retrieved products and synthesize a recommendation that accounts for all the user's stated constraints, rather than returning a single best-match product.
